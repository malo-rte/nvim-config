== Abstract

This document is the *shared Nerd Font icon system for the dev-tools suite*. It
fixes how the suite's user interfaces choose the glyph for an object and how they
render the independent status overlays that sit beside it. It is a reference: the
tables are the specification.

The system spans three parts:

. *File manager icons* -- the primary type icon for a filesystem object plus the
  filesystem, permission, Git, and backup status overlays shown beside it.
. *Status application icons* -- the module glyphs a status bar such as Waybar
  shows for workspaces, clock, CPU, memory, battery, audio, network, and the
  other system-state modules.
. *Cross-application design rules* -- the semantics, precedence, color,
  accessibility, versioning, override, and configuration policy every consumer
  shares so one meaning maps to one glyph everywhere.

The system is suite-wide, not tied to one program. Its consumers are the `wfm`
workspace file manager
(xref:../../architecture/DEV-TOOLS-ARCH-0012-file-manager/DEV-TOOLS-ARCH-0012-file-manager.adoc[DEV-TOOLS-ARCH-0012]),
the maintainer's Neovim configuration, the Waybar status bar, and the
filetype-to-icon exporters that emit configurations for third-party tools. Every
consumer resolves the same rule data, so a file shows the same icon in the file
manager, the editor, and a directory listing, and a system state shows the same
icon in the file manager and the status bar.

== Introduction

A dev-tools user interface shows an object -- a file, or a system-state module --
with a primary glyph and, in separate positions, a set of status overlays. This
reference specifies both, for file managers and for status applications, and the
design rules common to both.

For a filesystem object it separates five independent layers, each resolved on
its own:

. *Primary type icon* -- the object's kind, filename, extension, or detected
  content.
. *Filesystem-object state* -- symlink, mount point, hardware or device node.
. *Permission and diagnostic state* -- unreadable, broken link, operation error.
. *Version-control state* -- per-file Git status and repository chrome.
. *Backup state* -- inclusion in a backup set and freshness.

The layers do not overwrite one another. The type icon names *what the object
is*; the status overlays name *what is currently true about it*. A consumer that
shows all five reserves a column per status layer; a consumer with less width
drops whole columns (<<_recommended_tui_layout>>) rather than merging their
meanings into the type icon. The status-application part applies the same
separation: a module's icon names the resource, and its state (warning, error,
disconnected) is a distinct overlay drawn from one shared semantic vocabulary
(<<_78_common_alert_vocabulary>>, <<_86_semantic_consistency>>).

Glyph identifiers in this document use Nerd Font class names
(`nf-md-*`, `nf-oct-*`, `nf-cod-*`, `nf-fa-*`, `nf-seti-*`, `nf-custom-*`). The
exact code point behind a name varies between Nerd Font releases, so a consumer
pins a glyph mapping version and treats the rule data as external configuration,
never as compiled-in constants (<<_91_nerd_font_version_policy>>,
<<_94_user_override_model>>).

== Scope

In scope: the rule precedence; the icon assignments for filesystem objects,
directories, exact filenames, compound and simple extensions, and detected
content; the four file-manager status-overlay sets; the status-application module
icons and their states (Waybar and comparable bars); the recommended layouts for
full, compact, and minimal widths; the matching algorithm; extension
normalization; icon-width handling; the ASCII fallback sets; the shared semantic,
color, accessibility, versioning, override, validation, and configuration
policy; and the recommended defaults.

Out of scope: the requirements a consuming tool must meet and the structure of
any one consumer (those live in that tool's own requirement and architecture
specs, for example
xref:../../architecture/DEV-TOOLS-ARCH-0012-file-manager/DEV-TOOLS-ARCH-0012-file-manager.adoc[DEV-TOOLS-ARCH-0012]);
the mechanism that detects Git status, reads system sensors, or computes backup
inclusion; the byte-level content-detection engine beyond the extents this
reference bounds; and the specific values of a color theme, which supplements
icons but is defined elsewhere.

== How to read this reference

Each numbered section is a rule group. Most groups are a table: the left columns
name the object, state, module, or pattern being matched, and the right columns
give the glyph and its Nerd Font class name. Prose refers to a group by its
number (``§45`` means <<_45_matching_algorithm>>). Where a section fixes an
algorithm or a policy -- precedence (<<_1_rule_precedence>>), matching
(<<_45_matching_algorithm>>), normalization (<<_46_extension_normalization>>),
state precedence (<<_87_state_precedence>>), version policy
(<<_91_nerd_font_version_policy>>) -- the rules are normative for a consumer that
claims to implement this system; the specific glyph choices are recommendations a
consumer may override through its configuration (<<_94_user_override_model>>).

The reference is one document; <<_111_recommended_document_organization>> notes
how it may later be split into semantics, file-manager, and status-application
parts for maintenance without changing the rules.

== 1. Rule precedence

Evaluate icon rules in this order:

[%autowidth, options="header"]
|===
| Priority | Rule source | Example
| 1 | Special filesystem object | Socket, FIFO, block device
| 2 | Broken symbolic link | Link target does not exist
| 3 | Exact filename | `Makefile`, `Cargo.toml`, `.gitignore`
| 4 | Compound extension | `.tar.gz`, `.d.ts`, `.dtso`
| 5 | Simple extension | `.rs`, `.pdf`, `.png`
| 6 | Detected MIME or content | Text, image, archive
| 7 | Executable bit or binary inspection | ELF executable, script
| 8 | Generic fallback | Unknown file
|===


Do not replace the primary type icon with Git, backup, permission, or error icons. Those states should use separate columns.

== Git rules

The Git glyphs draw from more than one Nerd Font class, so one class is fixed
*per concept* -- every consumer resolves the exact code point listed in the
tables below, no matter which suite it borrows from:

* *Per-file diff states* (<<_35_per_file_git_state>>) use the Codicon
  `nf-cod-diff_*` set, with `nf-cod-copy` for a copy, `nf-cod-git_merge` for an
  unmerged conflict, `nf-cod-file_submodule` for a submodule, and the Font Awesome
  `nf-fa-question` for untracked (Codicon has no distinct untracked glyph).
* *Repository-level indicators* (<<_37_repository_level_git_indicators>>) use
  Octicons `nf-oct-*` where one exists (repository, alternative branch, stash),
  the Powerline `nf-pl-branch` for the current branch, the Codicon `nf-cod-git_*`
  set for the concepts Octicons lack a clean glyph for (commit, merge, compare,
  fetch, pull request), and `nf-fa-tag` for a tag; ahead/behind stay ASCII
  (`↑`/`↓`).

The chosen code points are the ones in the tables; they are the single source of
truth for the `wfm` git column and repository chrome and for any other consumer.

== Implementation policy

=== 45. Matching algorithm

[%autowidth, options="header"]
|===
| Step | Action
| 1 | Read `lstat()` metadata without following links
| 2 | Classify special filesystem objects
| 3 | Resolve valid symlink target type when configured
| 4 | Match exact basename rules case-sensitively
| 5 | Match case-insensitive filename rules where appropriate
| 6 | Match longest compound extension
| 7 | Match simple extension
| 8 | Inspect magic bytes for ambiguous or unknown types
| 9 | Detect text and shebang
| 10 | Apply generic fallback
| 11 | Calculate independent Git state
| 12 | Calculate independent backup state
| 13 | Add permission and diagnostic indicators
|===


=== 46. Extension normalization

[%autowidth, options="header"]
|===
| Rule | Example
| Prefer longest suffix | `file.tar.zst` before `.zst`
| Preserve case when meaningful | `Makefile`, `Image`, `Kconfig`
| Otherwise compare extensions case-insensitively | `.JPG` equals `.jpg`
| Treat leading-dot names as filenames first | `.gitignore` is not extension `gitignore`
| Support numeric shared-library suffixes | `libfoo.so.3.2`
| Support generated compression suffixes | `.dtb.gz`, `.ko.zst`
| Permit project-local overrides | `.bin` as firmware in embedded project
|===


=== 47. Icon-width handling

[%autowidth, options="header"]
|===
| Requirement | Recommendation
| Cell width | Measure rendered width rather than assuming one cell
| Padding | Reserve two terminal columns for the icon field
| Variation selectors | Avoid unless explicitly supported
| Combining characters | Do not use for status overlays
| Missing glyph | Fall back to ASCII or generic icon
| Font requirement | Require a Nerd Font 3.x-compatible font
| Configuration | Allow all glyphs to be overridden
| Testing | Test with Kitty, foot, Alacritty, and other common terminals
|===

[%autowidth, options="header"]

== Waybar and status application icon recommendations

This section defines a compact, consistent icon vocabulary for Waybar, eww, i3status-rust, polybar, tmux status lines, notification daemons, and similar desktop status applications.

The main design rule is to use one semantic family per module and reserve warning icons for exceptional states. Normal states should remain visually quiet.

=== 50. General status-bar principles

[%autowidth, options="header"]
|===
| Principle | Recommendation
| Normal state | Use a neutral icon with no warning decoration
| Warning state | Use a warning or alert variant only when action is needed
| Critical state | Use a strong alert, error, or disconnected icon
| Numeric value | Place the value after the icon
| Disabled state | Prefer a slashed or off variant
| Unknown state | Use a question or help-circle icon
| Module width | Keep icons to one terminal cell where possible
| Animation | Avoid changing icons rapidly unless showing progress
| Colour | Use colour as reinforcement, not the sole indicator
| Consistency | Do not use the same icon for unrelated modules
|===


A typical status item should use this form:

----
<icon> <value>
----

Examples:

----
󰍛 12%
󰖩 1 Gbit/s
󰕾 48%
󰂄 83%
----

== Cross-application icon design guidelines

This section defines the behavioral rules that apply to both TUI file managers and desktop status applications. It complements the icon lookup tables by specifying how icons should be selected, combined, displayed, themed, and overridden.

=== 86. Semantic consistency

Each icon should have one primary meaning throughout the application.

[%autowidth, options="header"]
|===
| Rule | Recommendation
| One icon, one meaning | Do not use the same icon for unrelated concepts
| Stable meaning | An icon should retain its meaning across views and modules
| Related states | Use variants from the same icon family where possible
| Normal state | Prefer a neutral icon or no icon
| Exceptional state | Use warning, error, disabled, or disconnected variants
| Context labels | Use tooltips or text when an icon may be ambiguous
| Avoid decoration | Do not add icons solely for visual variety
|===


Recommended shared meanings:

[%autowidth, options="header"]
|===
| Icon | Meaning
| 󰄬 | Successful, current, verified, or complete
| 󰥔 | Pending or waiting
| 󰑐 | Synchronizing, refreshing, or updating
|  | Warning
|  | Error
| 󰅚 | Critical failure
| 󰋗 | Unknown or unavailable
| 󰅖 | Disabled or explicitly off
| 󰖪 | Disconnected
|  | Locked or inaccessible
|  | Link or association
|===


Do not use `󰄬` for every healthy state. Excessive checkmarks create visual noise and reduce the usefulness of success indicators.

=== 87. State precedence

When several states apply to one display position, use the highest-priority state.

[%autowidth, options="header"]
|===
| Priority | State class | Examples
| 1 | Critical failure | Integrity failure, hardware failure, unrecoverable error
| 2 | Broken or inaccessible object | Broken symlink, permission denied, missing device
| 3 | Error | Failed operation, backup failure, service failure
| 4 | Warning | Low battery, low disk space, high temperature
| 5 | Active operation | Copying, scanning, syncing, updating
| 6 | Modified or stale state | Git modified, backup outdated, configuration changed
| 7 | Disabled or disconnected | Muted, offline, Bluetooth disabled
| 8 | Normal semantic type | File type, network type, power source
| 9 | Decorative context | Category, application branding
|===


Separate columns should retain independent state. For example:

[%autowidth, options="header"]
|===
| Type | Git | Backup | Permission
| Broken symlink | Modified | Excluded | Inaccessible
|===


The primary icon should still indicate the broken symlink. Git, backup, and permission state should remain in their own columns.

=== 88. Overlay and composition policy

Avoid combining several glyphs into one terminal cell.

[%autowidth, options="header"]
|===
| Technique | Recommendation
| Combining characters | Avoid
| Zero-width overlays | Avoid
| Emoji modifiers | Avoid
| Icon plus superscript | Use only if width is verified
| Adjacent status columns | Preferred
| Icon plus text suffix | Acceptable
| Color-only overlay | Insufficient on its own
|===


Recommended composition:

----
T  G  B  P  Name
    link
----

Avoid:

----
⃠
----

The latter may render differently between terminals and fonts.

=== 89. Color semantics

Icons should remain meaningful without color. Color should reinforce, not replace, state.

[%autowidth, options="header"]
|===
| Semantic role | Suggested meaning
| Default foreground | Normal state
| Dim foreground | Inactive, ignored, or secondary state
| Accent | Selection or focused state
| Success | Completed or verified
| Information | Informational state
| Warning | Attention required
| Error | Failed operation
| Critical | Immediate action required
| Disabled | Explicitly off or unavailable
| Modified | Changed or stale state
|===


Applications should use theme role names rather than hard-coded RGB values.

Example theme roles:

[source,toml]
----
[colors]
normal = "foreground"
dim = "subtle"
accent = "accent"
success = "success"
info = "info"
warning = "warning"
error = "error"
critical = "critical"
disabled = "disabled"
modified = "modified"
----

The icon configuration should refer to roles:

[source,toml]
----

icon = ""
color = "modified"

[backup.failed]
icon = "󰅚"
color = "error"
----

=== 90. Accessibility requirements

Important information must not be represented only by an icon or only by color.

[%autowidth, options="header"]
|===
| Requirement | Recommendation
| Text alternative | Provide tooltip, label, or status description
| ASCII fallback | Required for every important state
| Monochrome support | All states must remain distinguishable
| High contrast | Support a stronger warning and selection theme
| Screen readers | Expose semantic text where the UI framework permits
| Color blindness | Do not rely on red versus green alone
| Icon ambiguity | Add a label or tooltip
| Motion sensitivity | Allow animation to be disabled
|===


Examples:

----
󰂃 8% battery
----

----
 Backup failed
----

An icon-only display is acceptable only for familiar, low-risk controls such as play, pause, lock, or power, and should still have a tooltip.

=== 91. Nerd Font version policy

The application should state and enforce a supported Nerd Font version range.

[%autowidth, options="header"]
|===
| Item | Recommendation
| Minimum version | Nerd Fonts 3.x
| Glyph source | Pin to a known release of `glyphnames.json`
| Code points | Do not silently update between application releases
| Migration | Document renamed or moved glyphs
| Missing glyph | Fall back to ASCII or a generic icon
| User override | Always allow replacement
| Font check | Optional diagnostic command or settings page
|===


Suggested metadata:

[source,toml]
----
[icon_set]
name = "nerd-fonts"
minimum_version = "3.0"
mapping_version = "3.4.0"
fallback = "ascii"
----

Avoid depending on glyphs that exist only in unreleased or development versions.

=== 92. Unicode and terminal width handling

Private-use glyphs do not always have consistent width across terminals and fonts.

[%autowidth, options="header"]
|===
| Requirement | Recommendation
| Width measurement | Use terminal cell-width calculation
| Default field width | Reserve two cells for icon columns
| Ambiguous-width glyphs | Test explicitly
| Variation selectors | Avoid
| Emoji presentation | Avoid
| Combining characters | Avoid
| Grapheme segmentation | Treat each configured icon as a grapheme
| Alignment | Pad after the icon, not before
| Truncation | Never split an icon grapheme
| Remote sessions | Support an ASCII-only mode
|===


A robust renderer should support:

----
icon_width = auto
icon_column_width = 2
----

Possible configuration:

[source,toml]
----
[rendering]

icon_column_width = 2
missing_glyph_policy = "ascii"
emoji = false
variation_selectors = false
----

=== 100. Performance and caching

Icon lookup should never become a visible file-manager bottleneck.

[%autowidth, options="header"]
|===
| Area | Recommendation
| Exact filenames | Hash map
| Extensions | Hash map or suffix trie
| Compound extensions | Longest-suffix trie
| Patterns | Precompiled glob set
| MIME detection | Lazy and bounded
| Magic detection | Cache by inode, size, and modification time
| Git status | Batch query per repository
| Backup status | Batch query per backup root
| Directory icons | Cache by basename and context
| Rendering | Cache resolved icon and width
| Configuration reload | Atomic replacement
|===


Avoid running `file(1)` separately for every entry. Use an in-process magic database or batched worker pool when content inspection is required.

=== 108. Localization policy

Icons should not depend on language-specific wordplay or culturally narrow metaphors.

[%autowidth, options="header"]
|===
| Requirement | Recommendation
| Labels | Keep all labels translatable
| Tooltips | Use localized full descriptions
| Units | Respect locale preferences
| Dates | Respect locale and user format
| Directionality | Test right-to-left layouts when supported
| Abbreviations | Avoid hard-coded English abbreviations where possible
| Icon meaning | Prefer globally recognizable concepts
| ASCII fallback | Permit localized text fallback
|===


The icon itself should remain stable while the associated label changes with locale.

=== 110. Reference glossary schema

A generated glossary should contain one row per glyph.

[%autowidth, options="header"]
|===
| Field | Description
| Semantic key | Stable application identifier
| Glyph | Rendered character
| Nerd Font name | Canonical glyph name
| Unicode code point | Numeric code point
| Meaning | Primary semantic meaning
| Category | File type, state, module, action
| Width | Measured cell width
| ASCII fallback | Non-Nerd-Font replacement
| Minimum version | First supported Nerd Font version
| Deprecated | Whether the mapping should be replaced
| Notes | Ambiguity or usage restrictions
|===

[glossary]
== Glossary

[glossary]
Nerd Font:: A patched font that adds a large set of developer and UI glyphs in
the Unicode private-use area. Glyphs are named by class (`nf-md-*` Material
Design, `nf-oct-*` Octicons, `nf-cod-*` Codicons, `nf-fa-*` Font Awesome,
`nf-seti-*` Seti-UI, `nf-custom-*`), and the code point behind a name can differ
between Nerd Font releases (<<_91_nerd_font_version_policy>>).

Status overlay:: A glyph shown in a dedicated position beside the primary icon,
reporting a current condition of the object -- filesystem, permission, Git, or
backup state for a file; warning, error, or disconnected state for a status
module -- rather than its kind. Status overlays are independent of the primary
icon and of each other.

Status module:: A single unit of a status application (such as a Waybar bar):
one resource or system state -- clock, CPU, battery, network -- with its own icon
and its own state overlay drawn from the shared semantic vocabulary.

Semantic vocabulary:: The small set of state meanings (success, pending, syncing,
warning, error, critical, unknown, disabled, disconnected) that every consumer
maps to one glyph each (<<_78_common_alert_vocabulary>>,
<<_86_semantic_consistency>>), so a state reads the same across applications.

Compound extension:: A multi-part file suffix matched as a unit, such as
`.tar.gz`, `.d.ts`, or `.dtb.gz`. Compound extensions are matched before the
trailing simple extension (<<_1_rule_precedence>>, <<_46_extension_normalization>>).

Magic bytes:: The leading bytes of a file that identify its format independently
of its name, read within the bounded window defined in <<_41_magic_and_content_rules>>
when name-based rules do not resolve.
